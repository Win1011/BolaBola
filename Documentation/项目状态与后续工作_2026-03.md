# BolaBola 项目状态与后续工作（2026 年 3 月）

本文汇总近期为解决 **会员/证书限制、同步、语音、分析** 等做的改动，以及**已知技术债**与推荐优化方向。更细的 App Group 说明见同目录 [`app_group_removal_and_restore.md`](./app_group_removal_and_restore.md)；手表安装问题见 [`watch_app_install_troubleshooting.md`](./watch_app_install_troubleshooting.md)。  
**iPhone 产品需求（三板块 + 双数值）** 见 [`PRD_iPhone_三板块_2026-04.md`](./PRD_iPhone_三板块_2026-04.md)；手表 v1 范围仍见 [`BolaBola Watch App/Documentation/PRD_v1.md`](../BolaBola%20Watch%20App/Documentation/PRD_v1.md)。

## 1. App Group：去掉又加回来的背景

- **动机**：Personal Team 等场景下，entitlements 里声明了 App Group，但 provisioning profile 未包含对应 group 时，可能导致 **手表伴侣 App 安装失败**。曾尝试 **从 entitlements 移除 App Group**，让 iPhone / Watch 各用 `UserDefaults.standard`，避免 profile 与声明不一致。
- **代价**：无 group 时，两端 **不共享** 同一套 `UserDefaults`；陪伴「游戏状态」等依赖共享 defaults 的能力，改为主要靠 **WatchConnectivity**（如手表 → iPhone 的 `csV1` 快照）补偿。
- **当前**：若已恢复 entitlements 中的 App Group 且账号侧能力配齐，则 `BolaSharedDefaults.resolved()` 会再次优先使用 `group.com.gathxr.BolaBola`。
- **后续**：在 `AppGroupConfig` / `BolaSharedDefaults` 中搜索 **`RESTORE_APP_GROUP_WHEN_PAID_DEV`** 可定位相关注释；续费正式开发者后应 **统一 entitlements、Developer Portal 里的 App Group、与 profile**，并做一次 **双端读写 + WC** 回归。

## 2. WatchConnectivity 与「LLM 配置」同步

- **设计**：iPhone 在 Keychain 中保存对话 API（Key / Base URL / 模型等），通过 `transferUserInfo` 同步到手表 Keychain（`ingestLLMConfigurationIfPresent`）。
- **问题**：同步链路不稳定或时序问题时，手表会出现 **`loadFromKeychain: missing API key`**，语音直连智谱无法发起。
- **已加补偿**：
  - 手表在 WC 激活后、以及伴侣 App 状态变化时，若本地无 Key，会发 **`requestSync = llmKeychain`**，请 iPhone 再执行 `pushStoredLLMConfigurationToWatchIfConfigured()`。
  - 日志（subsystem `com.gathxr.BolaBola.sync`，category `WatchConnectivity`）中可搜 `ingestLLMConfiguration`、`pushStoredLLMConfigurationToWatchIfConfigured skip`。
- **后续优化**：恢复可靠同步后，可弱化或保留「拉取」作兜底；考虑在设置页增加「仅向手表推送配置」按钮；长期应用 **登录账号 + 云端下发密钥** 替代明文 WC 传 Key。

## 3. API / LLM：当前是「多来源、需统一」

| 来源 | 用途 | 说明 |
|------|------|------|
| **Keychain**（iPhone 设置里保存） | 主路径 | 优先；可同步到表。 |
| **`Shared/LLM/LocalLLMDevSecrets.swift`** | 开发兜底 | Keychain 无有效 Key 时使用；**`apiKey` 提交仓库时应为空**，本地自填，**勿向公开仓库推送含真 Key 的提交**。 |
| **智谱 `open.bigmodel.cn`** | 手表录音 ASR + 对话 | `LLMClient.transcribeAudio` 仅支持智谱域；Base URL 需为 `…/api/paas/v4` 类路径。 |
| **默认 OpenAI 地址** | 未保存 Base URL 时的默认 | 与智谱 Key 混用会 401；设置里务必保存正确 Base URL。 |

- **日志**：手表语音调试可过滤 category **`WatchVoice`**（`transcribeAudio`、`replyFromAudio`、`toggleMic` 等）。
- **后续优化**：统一「单一配置源」策略（仅 Keychain 或仅云端）；上架前移除或条件编译 `LocalLLMDevSecrets`；ASR 与 chat 的模型字段可拆分配置。

## 4. Firebase（Analytics）

- 已通过 **Swift Package Manager** 引入 **Firebase iOS SDK**，iOS 目标链接 **`FirebaseAnalyticsCore`**（无 IDFA 路径）；**手表目标未链 Analytics**（二进制产物不支持 watchOS）。
- 根目录 **`GoogleService-Info.plist`** 已加入 **iOS + Watch** 的 Copy Bundle Resources（当前主要为 iOS 使用）；若 Watch 单独注册 Firebase 应用，需控制台单独配置。
- 初始化：`FirebaseApp.configure()` 在 **`IOSAppDelegate`**（`BolaBola iOS/App/IOSNotificationRouter.swift`）的 `didFinishLaunching` 中调用。
- **后续**：按需加 Crashlytics / Remote Config；Watch 若要用 Firebase 非 Analytics 能力，按官方文档单独选 product。

## 5. 手表语音链路（简要）

1. 手表 **麦克风** → `WatchSpeechRelay`（录音 WAV）→ **`ConversationService.replyToUserFromRecordedAudio`** → 智谱 **ASR** → **chat** → `PetViewModel` 播放回复。
2. 云端 ASR 失败时，仍会尝试 **iPhone 中继转写**（`transferFile` + `IOSpeechRelayTranscriber`），不依赖手机时的优化见第 3 节（配置与网络）。
3. `BolaSpeechCapture` 在 watchOS 上为 **无 Speech 框架的占位**，真正识别不走系统 Speech。

## 6. 工程目录整理（2026-03）

为侧边栏清晰，代码已按职责分子文件夹（**不**把超大文件硬合并）：

- **`Shared/Defaults/`**：共享 UserDefaults / App Group 相关、Keychain 助手、陪伴持久化键。
- **`Shared/Sync/`**：`BolaWCSessionCoordinator`、`WCSyncPayload`、聊天记录模型与合并逻辑。
- **`Shared/LLM/`**：`LLMClient`、`LLMModels`、`ConversationService`、`LocalLLMDevSecrets`。
- **`Shared/Companion/`**：`CompanionTier`。
- **`Shared/Reminders/`**、**`Shared/Digest/`**：提醒与每日总结相关。
- **`BolaBola iOS/App/`**、**`Features/*`**、**`Design/`**：入口、功能分区、主题与 mock。
- **`BolaBola Watch App/App/`**、**`Views/`**、**`Speech/`**、**`Pet/`**、**`Health/`**、**`Reminders/`**、**`Documentation/`**。

手表端 **`WatchSpeechRelayRecorder` + `WatchSpeechRelayCapture`** 合并为同一文件 **`Speech/WatchSpeechRelay.swift`**（两类型仍分开，总行数适中，便于阅读）。

Xcode 使用 **File System Synchronized** 的文件夹，磁盘上移动文件后工程会自动收录，**一般无需改 `project.pbxproj`**。

## 7. 建议的后续优先级（备忘）

1. **同步与配置**：稳定 WC + Keychain 或改为账号 + 云端密钥；减少对手写 `LocalLLMDevSecrets` 的依赖。  
2. **App Group**：与付费开发者账号对齐后，全文测 **共享 defaults + 快照 + WC** 是否重复或冲突。  
3. **提醒功能**：在现有 Reminder / Health 模块上继续产品化（与语音、陪伴值联动等）。  
4. **安全与上架**：密钥管理、隐私政策、Sign in with Apple（若上登录）等。  
5. **结构**：新增文件时优先放入上面对应子目录，避免再回到扁平大列表。

## 8. 生活 Tab 布局、滚动边缘与 GitHub `main` 对齐说明

- **根级 `TabView`（iOS 26）** 使用 `tabBarMinimizeBehavior(.onScrollDown)` 时，系统会把 **当前 Tab 的主纵向 `ScrollView`** 与底栏收起关联。
- **若用横向分页 `ScrollView` 或内层分页 `TabView`（`.page`）包住「生活 / 时光」**：容易出现 **纵向滚动时底栏不再收起**，且分页容器可能 **改变安全区**、与 Figma 全幅背景不一致。当前采用 **`switch(lifeSegment)` + 两列各自纵向 `ScrollView`**，由导航栏 `IOSLifeSegmentLarge` 切换，无左右滑。
- **滚动边缘**：根视图与各页通过 `scrollEdgeEffectStyle`（见 `IOSGlassChrome.swift`）统一顶/底质感；与 [Win1011/BolaBola](https://github.com/Win1011/BolaBola) 上 `IOSRootView` / `IOSGlassChrome` 若一致，差异多半在 **`IOSLifeContainerView` 的滚动层级**与功能迭代（记录类型、天气卡等），而非神秘「UI 漂移」。
- **底栏 `TabSection` 与「对话」之间的间距、三格胶囊宽度**：由系统 Liquid Glass Tab 绘制，**无稳定公开 API** 微调；导航栏「生活｜时光」标题区可单独压字号/间距（`IOSLifeToolbarCenter.swift`）。
